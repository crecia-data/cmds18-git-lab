# 04. 검토 관문(PR) — 바로 반영 말고, 보고 승인

> **한 줄 목표**: 변경을 곧장 반영하지 않고 **PR로 올려 사람이 검토·승인**하는 흐름을 체험한다. ← **오늘 강의의 핵심**
> **소요**: 약 12분 · **앞 단계**: `/step 3`(실험 브랜치 있음) · **이번에 쓰는 것**: `/ship`

---

## 🤔 왜 이게 핵심일까?

AI는 빠르지만 **항상 옳지는 않습니다.** 그래서 AI가 만든 변경을 **무조건 본편(main)에 넣지 않고, 사람이 한 번 보고 도장을 찍는 관문**이 필요합니다. 그게 **PR(Pull Request)**입니다. *"AI에게 일을 맡기되, 통제권은 사람이 쥔다"* — AI 협업 시대의 가장 중요한 습관입니다.

## 💡 개념: PR = "결재 올립니다"

> - **PR(Pull Request)** = *"이 변경 반영해도 될까요?"* 하고 올리는 **결재 서류**.
> - **review/comment** = 서류에 다는 **빨간펜 피드백**.
> - **merge(머지)** = 승인 **도장** → 본편(main)에 정식 반영.
> - **issue** = 할 일 **포스트잇**(AI에게 줄 작업 메모).

---

## 🛠️ 따라하기

### ① 검토용으로 올리기 — `/ship`
(03에서 만든 실험 브랜치 상태에서)
```
검토받게 PR로 올려줘
```
**화면에서 일어나는 일**: Claude가 브랜치를 GitHub에 push하고 **PR을 생성**한 뒤 그 **PR 주소(URL)**를 알려줍니다.

### ② 뭐가 바뀌는지 보기
```
뭐가 바뀌는지 보여줘
```
Claude가 변경점(diff)을 사람 말로 요약합니다 — *"강조색이 빨강→파랑, 소개문구 1줄 추가."* PR 주소를 브라우저로 열어 직접 봐도 좋습니다(코멘트도 달아볼 수 있어요).

### ③ 승인 — 반영하기
```
좋아, 반영해줘
```
→ 머지됩니다. *"✅ 당신이 검토하고 승인해서 main에 반영됐어요."* **사람의 도장**이 찍힌 겁니다.

### (심화) 이슈를 작업 큐로
```
다음에 할 일로 '연락처 줄 추가'를 깃허브에 이슈로 적어둬
```
→ **issue = 할 일 포스트잇 / AI의 작업 큐.** 나중에 *"그 이슈 보고 처리해줘"*로 이어집니다.

## ⚠️ 흔한 실수 / FAQ
- **"main에서 바로 올리려니 막혀요"** → 정상입니다(가드레일). main 직접 push는 막아뒀어요. 먼저 `/experiment`로 브랜치를.
- **PR이 안 보여요** → gh 로그인·원격 저장소가 있는지 `/setup-check`.

## 🎤 강사 포인트
- **검증된 비교 슬라이드**: 같은 "AI가 PR, 사람이 리뷰"를 **Claude Code `@claude`** vs **GitHub Copilot cloud agent(세션 59분·draft PR)**로 비교 — 둘 다 **사람 승인 게이트는 동일**. 기획안 §2 표 활용.
